# a11y / interaction baseline

> Consolidates the a11y rules scattered across the code in one place. Keyboard navigation, focus management, ARIA attributes, and the visual feedback (cursor / focus ring) baseline. Per-component details cross-ref each component spec.

---

## 1. Scope

This rule applies to:

- Pages (`(dashboard)/*`, `(docs)/*`, `(api-reference)/*`, `(design-system)/*`)
- Domain components (`src/components/api-portal/*`)
- The *usage side* of shadcn primitives (a11y inside the primitive is shadcn's responsibility)

What this rule does not cover:
- Color contrast ratios (WCAG AA etc.) — separate rule / Figma verification
- Motion / animation (`prefers-reduced-motion`) — separate rule (currently undefined)
- Localization / text direction (RTL) — outside Phase1

## 2. Keyboard navigation

### 2.1 Tab order

Every interactive element must be reachable by Tab. Standard order:

1. TopNav menu items → Search → Account dropdown
2. Page body: Breadcrumb → Toolbar search → CTA → table row inline action → row ⋯ menu → Pagination
3. When a dialog opens: first focusable inside the dialog → ... → footer button → on close, return to the trigger

### 2.2 `focus-visible:ring` — standard

Show a visible ring only on keyboard focus (the ring is not shown on mouse focus):

```tsx
className="outline-none focus-visible:ring-2 focus-visible:ring-ring"
```

- `outline-none` — remove the browser default outline (replaced by focus-visible)
- `focus-visible:ring-2` — 2px ring on keyboard focus
- `focus-visible:ring-ring` — ring color token (semantic)

### 2.3 Keyboard shortcuts

Handled by the primitive by default:
- **Dialog / DropdownMenu / Tooltip**: Esc closes
- **DropdownMenu**: ↑↓ move items, Enter selects
- **Tabs**: ←→ switch tabs, Home/End first/last
- **Combobox / Select**: ↑↓ options, Enter selects, Esc closes

→ Since the primitive handles them, don't add an `onKeyDown` implementation at the page level.

## 3. Focus management

> **Visual tokens are in a separate doc**: ring width / color / aria-invalid conversion patterns are defined in [focus-ring.md](./focus-ring.md). This section focuses on *behavior* (autoFocus blocking, dialog focus restoration, sr-only absorption, etc.).

### 3.1 autoFocus blocking — all form-dialogs

When a dialog opens, no input field should receive automatic focus (preventing an Edit dialog from entering edit mode the instant it opens). 2-layer defense:

**(1) sr-only focus absorption span** — right below DialogContent:

```tsx
<DialogContent>
  <span tabIndex={0} className="sr-only outline-none" aria-hidden="true" />
  <DialogHeader>...</DialogHeader>
  ...
</DialogContent>
```

When the primitive looks for the first focusable node, this span is caught and absorbs it *with no visual focus*.

**(2) Explicit `autoFocus={false}` on the first input/select**:

```tsx
<Input id="..." autoFocus={false} ... />
```

Defensive — in some browsers sr-only alone may not be enough.

> Detailed rule: `patterns/form-dialog.md` §8

### 3.2 hover events on a disabled Button

`<Button disabled>` doesn't receive hover/focus events. To explain the reason with a Tooltip, wrap it:

```tsx
<Tooltip>
  <TooltipTrigger
    render={
      <span tabIndex={0} className="inline-block outline-none focus-visible:ring-2 focus-visible:ring-ring rounded-md">
        <Button disabled>Delete</Button>
      </span>
    }
  />
  <TooltipContent>...</TooltipContent>
</Tooltip>
```

> Detailed rule: "Disabled action reason" in `components/tooltip.md`

### 3.3 Dialog focus restoration

When a dialog closes, focus automatically returns to the trigger element — handled by the primitive. No extra implementation needed.

## 4. ARIA attributes

### 4.1 `aria-label` — required when there is no visual label

Interactive elements with *no visible text* — icon-only buttons, triggers, overlay links — require `aria-label`.

| Case | Format |
|---|---|
| ⋯ DropdownMenu trigger | `aria-label={`Actions for ${name}`}` |
| Avatar trigger (Account) | `aria-label="Account menu"` |
| Search icon button (TopNav) | `aria-label="Search"` |
| Dark mode toggle | `aria-label="Switch to dark mode"` / `"Switch to light mode"` |
| Overlay Link (TeamCard) | `aria-label={`View ${name} team`}` |
| Copy / Eye icon button (ViewApiKey) | `aria-label="Copy API key"` / `"Reveal API key"` |

> Detailed usage examples: `components/dropdown-menu.md`, `patterns/clickable-card-with-menu.md`

### 4.2 `aria-invalid` — form validation

Applied to Input / SelectTrigger when a required value is unmet:

```tsx
<Input aria-invalid={errors.email} ... />
<SelectTrigger aria-invalid={errors.role}>...</SelectTrigger>
```

The screen reader recognizes the invalid state. Paired with a visual Label `text-destructive` + error message.

> Detailed rule: `patterns/form-dialog.md` §5

### 4.3 `aria-current="page"` — current page

The last item of the Breadcrumb (the BreadcrumbPage primitive assigns it automatically):

```tsx
<BreadcrumbPage>API Keys</BreadcrumbPage>
// → aria-current="page" applied inside the primitive
```

Automatic when using the primitive — no separate writing needed.

### 4.4 `aria-describedby` — Tooltip / form help

The Tooltip primitive handles it automatically — TooltipTrigger automatically references the id of TooltipContent.

When there is form help text (no current use case, future):

```tsx
<Label htmlFor="email">Email</Label>
<Input id="email" aria-describedby="email-help" />
<p id="email-help" className="text-sm text-muted-foreground">We'll never share your email.</p>
```

### 4.5 `aria-hidden` — decorative elements

Purely visual elements are hidden from the screen reader:

| Case | Applied |
|---|---|
| Breadcrumb separator (ChevronRight) | `aria-hidden="true"` (primitive automatic) |
| Icon accompanying a text label (e.g. Settings + "Settings" text) | aria-hidden on the icon — the label conveys the meaning |
| sr-only focus absorption span | `aria-hidden="true"` — not interactive, conveys no information |

### 4.6 role / role branching (Toast)

The Sonner primitive assigns it automatically:
- success / info / warning → `role="status"` (passive announce)
- error → `role="alert"` (assertive announce — read immediately)

## 5. Visual feedback

### 5.1 Global cursor pointer (`globals.css`)

```css
button:not(:disabled):not([aria-disabled="true"]),
[role="button"]:not([aria-disabled="true"]) {
  cursor: pointer;
}
```

- A hover pointer cursor on all clickable button / `[role=button]`
- **disabled / aria-disabled excluded** — keep the default (`not-allowed` or `default`)
- No need to set cursor at the individual component level (the global rule handles it)

### 5.2 `cursor-default` — hover-only elements that aren't click actions

Info / Help icons that are Tooltip triggers:

```tsx
<Info className="h-3.5 w-3.5 text-muted-foreground cursor-default" />
```

→ Visually indicates that hovering only shows a tooltip with no click action (a non-pointer cursor prevents the "clickable" misunderstanding).

### 5.3 `hover` + `focus-within` pair

Consistent visual feedback for mouse hover ↔ keyboard focus:

```tsx
className="hover:bg-muted/30 focus-within:bg-muted/30"
```

- `hover:bg-muted/30` — mouse hover
- `focus-within:bg-muted/30` — same visual when anything inside is focused

→ Keyboard users get the same visual feedback as mouse users.

> Detailed rule: `patterns/clickable-card-with-menu.md` §4

## 6. Per-component a11y checklist

| Component / pattern | a11y rule |
|---|---|
| **Button** | Global cursor-pointer (globals.css). Auto-excluded when disabled. `aria-label` required when icon-only. |
| **Dialog** | Esc closes (primitive). Focus restoration (primitive). form-dialog's sr-only span (§3.1). |
| **DropdownMenu** | trigger `aria-label` required. ↑↓ keys (primitive). align="end" safe on the right of the screen. |
| **Tooltip** | TooltipProvider mounted at the root. delay=0. Wrap a disabled trigger in `<span tabIndex={0}>`. `cursor-default` icon. |
| **Toast (Sonner)** | role="status"/"alert" (primitive automatic). No toasts for permanent info (states.md §5). |
| **Breadcrumb** | aria-label="breadcrumb" (primitive). Last item BreadcrumbPage = aria-current="page" automatic. |
| **Input / Select** | `aria-invalid` + Label `text-destructive` pair. `htmlFor` ↔ `id` match. |
| **Table** | Sortable indication on SortableHead (lucide chevron icon). Row identifier in the Action ⋯ aria-label (`Actions for ${name}`). |
| **Sidebar (mobile Sheet)** | SheetHeader's `<SheetTitle className="sr-only">` (a11y label only, no visual). SidebarTrigger `aria-label`. |
| **form-dialog pattern** | sr-only focus absorption span + first input autoFocus={false}. |
| **clickable-card-with-menu** | overlay Link `aria-label`, focus-visible:ring, hover+focus-within pair. |

## 7. Anti-patterns

| ❌ | Reason |
|---|---|
| Missing `aria-label` on an icon-only button / trigger | The screen reader can't identify it. Can't tell whether it's clickable. |
| Applying only `outline-none` (without focus-visible:ring) | Keyboard users can't see the focus position. Always pair them. |
| Forcing `cursor-pointer` on a visual-only icon | Looks clickable. Info / Help icons use `cursor-default`. |
| Applying a Tooltip trigger directly to a disabled element | Disabled elements don't receive events. A `<span tabIndex={0}>` wrap is required. |
| Adding `cursor-pointer` per component, ignoring the global cursor rule | The global rule handles it. Redundant — and risks missing the disabled branch. |
| Visual-only error (`text-destructive`) without `aria-invalid` | The screen reader can't recognize the error. Pair visual + ARIA. |
| Custom keyboard shortcut (`onKeyDown=Esc to close`) the primitive already handles | Duplicate code. The primitive is more accurate. |
| Making the last Breadcrumb item a BreadcrumbLink (a link to itself) | Missing aria-current="page". Use BreadcrumbPage. |
| `hover` alone without `focus-within` | Visual feedback gap for keyboard users. Pair them. |
| A form dialog with only autoFocus={false} and a missing sr-only span | In some browsers the first input still gets focus. Both layers are needed. |

## 8. Pre-release manual checks (recommended)

When working on a new page / component, verify the following directly:

### Keyboard-only check
- [ ] All interactive elements reachable by Tab
- [ ] Tab order matches the visual order
- [ ] focus-visible ring shows on all focusable elements
- [ ] Esc closes dialogs / menus
- [ ] Enter / Space activates buttons / menu items

### Quick screen reader check
- [ ] Icon-only buttons are read with a meaningful label
- [ ] Form validation errors are announced
- [ ] The current page in the Breadcrumb is announced as "current page"

### Visual consistency
- [ ] hover / focus-within visual feedback pair
- [ ] Disabled elements have a not-allowed cursor (global rule)
- [ ] Tooltip triggers have cursor-default (indicating not a click)
